## User Scenarios & Testing *(mandatory)*

### User Story 1 - View Landing Page (Priority: P1)

As a new user, I want to see an engaging landing page so that I can understand the project and be encouraged to start reading the textbook.

**Why this priority**: The landing page is the entry point for users and crucial for initial engagement.

**Independent Test**: Can be fully tested by navigating to the root URL (/) and verifying the visual presentation and the functionality of the "Start Reading" button.

**Acceptance Scenarios**:

1. **Given** a user visits the root URL `/`, **When** the page loads, **Then** they see the "Classical Modern" design with a Hero image generated by `nano_banana`.
2. **Given** a user visits the root URL `/`, **When** the page loads, **Then** they see a "Start Reading" button.
3. **Given** a user is on the landing page, **When** they click "Start Reading", **Then** they are redirected to the Chapter 1 or Introduction page (`/docs/intro`).

### User Story 2 - Read Chapter Content (Priority: P1)

As a learner, I want to read the textbook chapters so that I can acquire knowledge about Physical AI & Humanoid Robotics.

**Why this priority**: Delivering core content is the primary goal of the textbook.

**Independent Test**: Can be fully tested by navigating to any chapter page and verifying its content, structure, and embedded visuals.

**Acceptance Scenarios**:

1. **Given** the user is reading Chapter 1, **When** they view the chapter, **Then** they see unique content relevant to "Introduction to Physical AI" with an image generated for "Evolution of AI timeline, diagram style".
2. **Given** the user is reading Chapter 2, **When** they view the chapter, **Then** they see unique content relevant to "Humanoid Robotics Basics" with an image generated for "Humanoid robot anatomy labels, schematic view".
3. **Given** the user is reading Chapter 3, **When** they view the chapter, **Then** they see unique content relevant to "Advanced Robotics Algorithms" including code blocks and flowcharts, with an image generated for "Path planning A* algorithm visualization".
4. **Given** the user is reading Chapter 4, **When** they view the chapter, **Then** they see unique content relevant to "Project Applications" focusing on real-world use cases, with an image generated for "Industrial robot arm assembly line, photorealistic".
5. **Given** the user is reading any chapter, **When** they reach the end of a major section, **Then** they see a 1-sentence micro-summary.
6. **Given** the user is reading any chapter, **When** they encounter code examples, **Then** the code blocks use proper syntax highlighting.
7. **Given** the user is reading any chapter, **When** they view embedded images, **Then** the images are stored in `/static/img` and referenced via relative paths.

### User Story 3 - Navigate Textbook (Priority: P2)

As a user, I want to easily navigate between different sections and chapters of the textbook so that I can find the information I need.

**Why this priority**: Good navigation is essential for usability, but secondary to the initial content availability.

**Independent Test**: Can be tested by verifying navbar links and sidebar functionality (on documentation pages).

**Acceptance Scenarios**:

1. **Given** a user is on any page, **When** they look at the Navbar, **Then** they see a Logo on the left and links for Home, Chapters, and GitHub on the right.
2. **Given** a user is on a documentation/chapter page, **When** they view the page, **Then** they see a sidebar.
3. **Given** a user is on the home page, **When** they view the page, **Then** they do NOT see a sidebar.

### Edge Cases

- What happens when an image fails to load? The system should display appropriate alt text and a placeholder.
- How does the system handle missing `nano_banana` instructions for an image? A default placeholder should be used, and a warning should be logged.
- What happens if the `nano_banana` tool fails to generate an image? The system should display a default placeholder and a warning should be logged.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST generate all textbook content (text, code examples, structure) using AI.
- **FR-002**: Manual intervention is ONLY permitted for file structure, navigation fixes, or critical corrections.
- **FR-003**: The system MUST prioritize the deployment of a Docusaurus website with high-quality chapters.
- **FR-004**: The content MUST strictly adhere to the "Physical AI & Humanoid Robotics" curriculum (ROS 2, Gazebo, NVIDIA Isaac, VLA). No generic AI content is allowed.
- **FR-005**: The system MUST present content in a "Classical Modern" UI/UX design.
- **FR-006**: The landing page (Home) MUST have a distinct UI/UX from the chapter pages.
- **FR-007**: The landing page MUST include a hero section with the headline "Mastering Physical AI & Robotics" and a sub-headline "A comprehensive, AI-generated curriculum for the next generation of engineers."
- **FR-008**: The landing page MUST feature a primary action button labeled "Start Reading" or "Explore Chapters" that navigates to `/docs/intro` (or the Chapter Index).
- **FR-009**: The system MUST use `nano_banana` to generate the hero image for the landing page featuring "Futuristic robot hand shaking human hand, oil painting style, high detail".
- **FR-010**: The landing page MUST include a 3-column grid highlighting key topics (Physical AI, Humanoids, Algorithms).
- **FR-011**: The Navbar MUST display a Logo on the left and links (Home, Chapters, GitHub) on the right.
- **FR-012**: A Sidebar MUST ONLY be visible on Documentation/Chapter pages.
- **FR-013**: The home page content MUST be promotional and introductory and MUST NOT contain actual chapter text.
- **FR-014**: All chapter content MUST reside strictly under the `/docs/` route.
- **FR-015**: The system MUST generate MDX files for all chapters.
- **FR-016**: Chapter 1 ("Introduction to Physical AI" at `/docs/physical-ai/introduction`) MUST include an image generated by `nano_banana` for "Evolution of AI timeline, diagram style".
- **FR-017**: Chapter 2 ("Humanoid Robotics Basics" at `/docs/humanoid-robotics/basics`, dependent on Chapter 1) MUST include an image generated by `nano_banana` for "Humanoid robot anatomy labels, schematic view" and a practical hands-on step.
- **FR-018**: Chapter 3 ("Advanced Robotics Algorithms" at `/docs/robotics-algorithms/advanced`, dependent on Chapters 1 & 2) MUST include an image generated by `nano_banana` for "Path planning A* algorithm visualization", and include code blocks and flowcharts.
- **FR-019**: Chapter 4 ("Project Applications" at `/docs/project-applications`, dependent on Chapters 1, 2 & 3) MUST include an image generated by `nano_banana` for "Industrial robot arm assembly line, photorealistic" and focus on real-world use cases.
- **FR-020**: The system MUST invoke the `nano_banana` tool whenever a section requires a diagram, hero image, or illustration.
- **FR-021**: The system MUST pass a descriptive prompt relevant to the chapter context to `nano_banana`.
- **FR-022**: The system MUST embed resulting images using standard Markdown syntax `![Alt Text](path/from/nano_banana)`.
- **FR-023**: The Docusaurus configuration MUST remove `src/pages/index.js` and replace it with the new Classical Modern Landing page code.
- **FR-024**: The Docusaurus routing MUST ensure the "Start Reading" button correctly links to the first chapter or sidebar introduction.
- **FR-025**: All content MUST use simple English and explain jargon immediately.
- **FR-026**: Content MUST use short paragraphs (3-4 sentences max) and frequent bullet points.
- **FR-027**: Every major section MUST end with a 1-sentence takeaway.
- **FR-028**: Word count for sub-chapters MUST target 400-600 words.
- **FR-029**: ROS 2 commands MUST be compatible with "Humble" or "Iron" distributions.
- **FR-030**: Hardware constraints (e.g., "This requires an NVIDIA GPU") MUST be explicitly mentioned where necessary.

### Key Entities *(include if feature involves data)*

- **Chapter**: Represents a section of the textbook with a title, route, content, and associated images.
- **Image**: Visual assets generated by `nano_banana`, embedded within chapters or on the landing page.
- **User**: An individual interacting with the textbook website.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% of the landing page UI elements (headline, sub-headline, button, hero image, feature grid) are rendered as specified.
- **SC-002**: Users can navigate from the landing page to the introduction chapter (`/docs/intro`) in under 2 seconds.
- **SC-003**: All four specified chapters are accessible via their defined routes and display content, images, and micro-summaries as per requirements.
- **SC-004**: All images in the chapters and on the landing page are successfully generated by `nano_banana` and displayed correctly.
- **SC-005**: The Docusaurus website builds and deploys to GitHub Pages without errors, rendering the new UI and content.
- **SC-006**: Average load time for chapter pages is under 3 seconds.
- **SC-007**: The navigation bar and sidebar function as designed, providing correct links and visibility based on the page type.
- **SC-008**: 100% of code examples within chapters use correct syntax highlighting.
