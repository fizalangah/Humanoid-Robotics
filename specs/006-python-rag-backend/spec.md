# Feature Specification: Python RAG Backend with UV

**Feature Branch**: `006-python-rag-backend`
**Created**: 2025-12-12
**Status**: Draft

## 1. Overview

This specification outlines the creation of a standalone Python backend to serve as the "Brain" for the Physical AI Textbook. The core function of this backend is to implement a Retrieval Augmented Generation (RAG) pipeline using Qdrant for the vector database, Cohere for embeddings, and OpenAI for generating final answers. The project will use `uv` for dependency management.

## 2. User Scenarios & Testing

### User Story 1: Contextual Chat Responses

As a user of the textbook, I want to ask questions to a chatbot and receive answers that are based on the content of the textbook, so that I can get accurate and relevant information.

**Acceptance Scenarios**:

1.  **Given** a user asks a question via the frontend chat interface,
    **When** the backend receives the query,
    **Then** the backend should return a textual answer generated by the RAG pipeline.
2.  **Given** a user asks a question about a specific topic covered in the documentation,
    **When** the backend processes the query,
    **Then** the answer should be consistent with the information present in the `docs` directory.

## 3. Requirements

### Functional Requirements

-   **FR-001**: The system MUST provide a `POST /chat` endpoint to receive user queries.
-   **FR-002**: The backend MUST use a `.env` file to manage all API keys (Qdrant, Cohere, OpenAI).
-   **FR-003**: The system MUST use the Cohere `embed-english-v3.0` model to generate vector embeddings for both the documentation and user queries.
-   **FR-004**: The system MUST store and retrieve document embeddings from a Qdrant Cloud vector database.
-   **FR-005**: The system MUST use an OpenAI model (GPT-4o or GPT-3.5) to generate the final answer based on the retrieved context.
-   **FR-006**: The system MUST include a data ingestion script (`ingest.py`) that recursively finds all `.mdx` files in the `../docs/` directory, cleans the text, generates embeddings, and uploads them to Qdrant.
-   **FR-007**: The chat API (`server.py`) MUST embed the user's query, search for the top 3 most relevant context chunks in Qdrant, and use them to generate an answer.
-   **FR-008**: The system prompt sent to the OpenAI model MUST be: "You are an AI Tutor for the Physical AI Course. Answer based strictly on the context provided."
-   **FR-009**: The backend MUST be a standalone Python application located in a new `backend` directory at the project root.
-   **FR-010**: The project MUST use `uv` for dependency management.

### Non-Functional Requirements

-   **NFR-001**: The chat API should respond to queries within a reasonable time frame (e.g., under 5 seconds).
-   **NFR-002**: The backend should handle errors gracefully (e.g., API failures, invalid requests).

## 4. Success Criteria

-   **SC-001**: A `backend` directory is created at the project root with the required file structure and dependencies.
-   **SC-002**: The `ingest.py` script successfully populates the Qdrant database with embeddings from the `.mdx` files.
-   **SC-003**: The `POST /chat` endpoint is functional and returns a text response.
-   **SC-004**: Chat responses are relevant to the user's query and grounded in the provided documentation context.
-   **SC-005**: All API keys are successfully loaded from a `.env` file and are not hardcoded.
