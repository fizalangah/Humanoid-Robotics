# Feature Specification: FastAPI Backend with LiteLLM & RAG

**Goal**: Build a Python backend using `fastapi` and `litellm` (Gemini 2.0 Flash) that answers questions based on the textbook content.
**Tooling**: Use `uv` for dependency management.

## 1. Directory Structure (`backend/`)
- `server.py`: The main API file (FastAPI + LiteLLM).
- `ingest.py`: Script to read `../docs/` and save embeddings to Qdrant.
- `.env`: Store API Keys (GEMINI_API_KEY, QDRANT_URL, QDRANT_API_KEY).

## 2. API Implementation (`server.py`)
- **Structure**: Use the user's provided snippet as a base but add RAG logic.
- **Libraries**: `fastapi`, `uvicorn`, `litellm`, `pydantic`, `python-dotenv`, `qdrant-client`, `cohere`.
- **Logic**:
  1. Receive `ChatRequest`.
  2. Embed the message using **Cohere**.
  3. Search **Qdrant** for relevant book context.
  4. Pass `Context + Question` to `litellm.completion` (Model: `gemini/gemini-2.0-flash`).
  5. Return response.

## 3. Ingestion Implementation (`ingest.py`)
- **Logic**: Read all `.mdx` files from `../docs`, embed them using Cohere, and upload to Qdrant Cloud.

## 4. Dependencies
- Run `uv init` and `uv add fastapi uvicorn litellm python-dotenv pydantic qdrant-client cohere`.

## User Scenarios & Testing

### User Story 1: Contextual Chat Responses with LiteLLM

As a user of the textbook, I want to ask questions to a chatbot and receive answers that are based on the content of the textbook, leveraging LiteLLM for diverse model access, so that I can get accurate and relevant information.

**Acceptance Scenarios**:

1.  **Given** a user asks a question via the frontend chat interface,
    **When** the backend receives the query,
    **Then** the backend should return a textual answer generated by the RAG pipeline using LiteLLM.
2.  **Given** a user asks a question about a specific topic covered in the documentation,
    **When** the backend processes the query,
    **Then** the answer should be consistent with the information present in the `docs` directory.

## Requirements

### Functional Requirements

-   **FR-001**: The system MUST provide a `POST /chat` endpoint to receive user queries.
-   **FR-002**: The backend MUST use a `.env` file to manage all API keys (GEMINI_API_KEY, QDRANT_URL, QDRANT_API_KEY).
-   **FR-003**: The system MUST use the Cohere `embed-english-v3.0` model to generate vector embeddings for both the documentation and user queries.
-   **FR-004**: The system MUST store and retrieve document embeddings from a Qdrant Cloud vector database.
-   **FR-005**: The system MUST use LiteLLM for `litellm.completion` with the `gemini/gemini-2.0-flash` model to generate the final answer based on the retrieved context.
-   **FR-006**: The system MUST include a data ingestion script (`ingest.py`) that recursively finds all `.mdx` files in the `../docs/` directory, cleans the text, generates embeddings, and uploads them to Qdrant.
-   **FR-007**: The chat API (`server.py`) MUST embed the user's query, search for the top 3 most relevant context chunks in Qdrant, and use them to generate an answer.
-   **FR-008**: The system prompt sent to the LiteLLM model MUST be: "You are an AI Tutor for the Physical AI Course. Answer based strictly on the context provided."
-   **FR-009**: The backend MUST be a standalone Python application located in a new `backend` directory at the project root.
-   **FR-010**: The project MUST use `uv` for dependency management.

### Non-Functional Requirements

-   **NFR-001**: The chat API should respond to queries within a reasonable time frame (e.g., under 5 seconds).
-   **NFR-002**: The backend should handle errors gracefully (e.g., API failures, invalid requests).

## Success Criteria

-   **SC-001**: A `backend` directory is created at the project root with the required file structure and dependencies.
-   **SC-002**: The `ingest.py` script successfully populates the Qdrant database with embeddings from the `.mdx` files.
-   **SC-003**: The `POST /chat` endpoint is functional and returns a text response.
-   **SC-004**: Chat responses are relevant to the user's query and grounded in the provided documentation context.
-   **SC-005**: All API keys are successfully loaded from a `.env` file and are not hardcoded.

## Assumptions

-   The `../docs/` directory contains valid `.mdx` files with relevant textbook content.
-   Valid API keys for Cohere, Qdrant, and Gemini (via LiteLLM) are provided in the `.env` file.
-   The Qdrant service is accessible from the backend.
